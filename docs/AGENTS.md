# 文档写作规范

本目录文档默认使用中文书写，服务于 TaskHive 的学习记录、阶段复盘和项目说明。修改 `docs/**` 下的 Markdown 前，先阅读并遵守本文件。

## 触发规则

- 新增、修改、润色 `docs/**` 文档时，必须遵守本文件。
- 新建或整理阶段笔记 `docs/notes/stage-*.md` 时，同时参考 `docs/notes/stage-template.md`。
- 只修改与当前任务直接相关的文档，不顺手重排全文或改写无关段落。

## 文档结构

阶段笔记统一放在 `docs/notes/`，文件名使用 `stage-XX-topic.md`，例如 `stage-00-toolchain.md`。新建阶段笔记时复制 `docs/notes/stage-template.md` 的结构，保留以下五个固定章节：

- `## 1. 本阶段目标`
- `## 2. 学到的知识点`
- `## 3. 关键决策（为什么这么做）`
- `## 4. 踩过的坑`
- `## 5. 遗留 TODO`

## 写作原则

- 面向“三个月后的自己”写：记录结论、关键理解、踩坑原因和解决方式。
- 先写结果，再写必要依据；避免空泛总结。
- 写决策时说明“选了什么、没选什么、为什么”；写踩坑时说明“现象、原因、解决方式”。
- 不编造未执行的命令、测试结果、tag、CI 状态或项目约定；不把临时想法写成事实。

## 符号规则

中文正文、标题、引用说明统一使用中文标点：`：`、`，`、`。`、`；`、`（ ）`、“ ”。

推荐：

- `阶段 0：工具链与项目骨架`
- `GOPATH（默认 ~/go）`
- `PATH，否则 go install 装的工具调不动。`

避免：

- `阶段 0:工具链与项目骨架`
- `GOPATH(默认 ~/go)`
- `PATH,否则 go install 装的工具调不动。`

## 代码与技术词

代码、命令、路径、配置键、函数名、变量名和精确输出用反引号包住。反引号内部保留原始英文符号，不做中文化。

示例：

- `go test -race ./...`
- `version: "2"`
- `t.Run(子测试名, ...)`
- `if got != want { t.Errorf(...) }`

## 格式约定

- 标题使用 Markdown `#` / `##`，阶段笔记标题格式为 `# 阶段 N：标题`。
- 列表项用 `-`，TODO 用 Markdown checkbox：`- [ ] ...`。
- 段落保持短句，优先清楚，不追求文学化表达。
- 不做无关格式化或批量改写，避免让文档 diff 变得难审。

## 审阅清单

- 中文正文是否统一使用中文标点。
- 代码、命令、路径、配置值内部是否保留原始英文符号。
- 阶段笔记是否保留模板的五个固定章节。
- TODO 是否使用 Markdown checkbox：`- [ ] ...`。
