# ADR-0001：任务命令使用 `sh -c` 执行

> 状态：Accepted
> 日期：2026-07-06
> 关联阶段：阶段 1

## 背景（Context）

`executor` 需要执行用户在 `Task.Command` 里提交的命令。这些命令可能包含 shell 特性：
管道（`echo a | grep a`）、重定向（`cmd > out.log`）、通配符（`rm *.tmp`）、
环境变量展开等。

Go 的 `exec.Command(name, args...)` 直接以“程序名和参数列表”方式启动进程，
**不经过 shell 解析**：它不会按空格拆分字符串，也不认识管道、重定向、通配符。
因此若直接用 `exec.Command`，用户提交的带 shell 特性的命令无法正常工作。

TaskHive 的定位是“用户自助提交任务的调度平台”，提交者本就是被信任的、
有权在系统上执行命令的角色。

## 决策（Decision）

`executor` 通过 `exec.Command("sh", "-c", task.Command)` 执行命令——
显式启动一个 shell，把整条命令交给它解析执行。这样 `task.Command` 可以使用完整的
shell 语法，与用户在终端里直接敲命令的体验一致。

## 备选方案（Alternatives）

- **直接 `exec.Command`，自己按空格拆分参数**：不启动 shell，自己把命令字符串 split 成参数。
  没选原因：无法支持管道、重定向、通配符等 shell 特性；简单空格拆分还会在带引号的参数上出错。
  对一个“执行任意任务”的调度器来说功能太受限。

- **要求用户提交结构化的程序和参数数组，不走 shell**：最安全，彻底杜绝注入。
  没选原因：用户体验差，常见的管道、重定向类任务无法表达；与“像在终端跑命令一样”的直觉不符。
  可作为未来面向不可信输入时的收敛方向。

- **命令白名单和参数校验**：只允许预先批准的程序。
  没选原因：当前是可信用户自助场景，白名单过度限制；维护成本高。留待多租户或不可信场景再引入。

## 后果（Consequences）

- 正面：支持完整 shell 语法，任务表达能力强，用户体验自然。实现简单。

- 负面和风险：等价于把用户输入交给 shell 执行，存在**命令注入**风险（用户能跑任意命令）。
  在“可信提交者”假设下这是可接受的——提交者本就有执行权限，不构成越权。
  但一旦系统开放给不可信输入（如公网多租户），此设计不再安全。

- 附带问题：`sh -c` 会 fork 子进程，导致超时杀进程在 Linux 上杀不干净
  （只杀父 `sh`、子进程变孤儿）。已知 bug，修复方向为进程组管理，记于 TODO，阶段 3 处理。

- 未来触发重新决策的条件：系统开放给不可信用户、引入多租户，或需要沙箱隔离。
  届时应转向“结构化参数不走 shell”或“白名单和沙箱”方案，并新开一篇 ADR 取代本篇。
